# Preyed Upon

Survival horror in Unity. Emergent, system-driven horror where AI behavior is dictated by biological eye placement (Predator vs. Prey). The player navigates, hides, and searches for clues in an abandoned, Chernobyl-inspired environment.

This README doubles as the project devlog — a running, plain-language log of where the game stands. No code details here — just what's working, what's next, and what to watch out for. See `Preyed_Upon_Detailed_Roadmap.md` for the full technical task breakdown.

---

## 2026-07-11 — Project kickoff, AI vision prototype

### Where we're at

We're at the very start of **Phase 1 (Prototype)**. The project itself is freshly scaffolded in Unity (URP), and the first real feature — the AI's vision system — has a working prototype. Everything is still built from primitive shapes (capsules, cubes); no real art, animation, or levels yet.

### Done

- **Unity project set up** (Universal Render Pipeline), pushed to GitHub.
- **AI Vision System (first pass):** AI characters can now be configured as either "Predator" or "Prey," and each behaves like a real animal eye-wise:
  - **Predators** see a narrower slice of the world directly ahead, but most of what they see is in stereo (both eyes), so they judge distance/position accurately.
  - **Prey** see almost all the way around themselves, but only a thin wedge directly in front is seen by both eyes — everything to the sides is "sensed" but not accurately placed (matches the real-world predator/prey eye diagram the team referenced).
  - Walls and obstacles correctly block the AI's line of sight (tested with placeholder wall/capsule objects).
- **Editor tools** so anyone can spin up a test scene with one click (`Preyed Upon` menu in Unity) to see the vision system working without building it by hand.
- **Player controller (first-person, first pass):** the player can walk, sprint, idle, crouch, and lie prone (crouch/prone are toggles, not holds), and lean left/right to peek around corners without exposing their whole body. Standing back up is blocked if there isn't physically enough headroom (won't stand up through a low ceiling or table). Movement speed and stance already feed into a "how much noise am I making right now" value, ready for the AI's hearing system whenever that gets built.
- **Editor tools** for the player too — one click builds a working, playable capsule-based player rig wired up with the above.
- **Hiding volumes:** trigger zones the player can walk into to become completely undetectable to any AI — even if they're still standing in the AI's cone with nothing physically blocking the view. Works by fully overriding the vision check, not by relying on occlusion, so it'll behave the same whether the "hiding spot" is a shadow, tall grass, or a locker later on. Supports overlapping volumes correctly (you only become visible again once you've left all of them).
- **A one-click way to drop a Predator into the test scene** watching the playable Player, so the eye mechanics and the hiding volume can be verified together instead of only against a stationary dummy target.
- **Buildings now sit flush with the ground.** Every building (reactor, both settlement types, checkpoint booths, ruins) gets its own leveled foundation pad carved into the terrain before it's placed, sized to cover its full footprint — previously only roads/river/pond got this treatment, so a building on a slope could've had corners floating above the ground or clipped into it.
- **Wilderness environment blockout, now a seeded zone generator:** one menu command builds a full, randomized Chernobyl-Exclusion-Zone-style map from a seed — a different layout every time you run it. Modeled directly on a reference map the team shared: an irregular fenced zone boundary with two checkpoint gates, one reactor complex (with its own perimeter fence), one large dense "Pripyat-style" settlement plus two smaller villages, a road network connecting all of it (so nothing is ever stranded — every settlement, the reactor, and both checkpoints are reachable), a winding river plus a separate pond, street lights and wrecked cars along the roads, scattered ruins and loot containers (lockers, weapons crates, safes, bags, a dumpster), a cave entrance marker, and forest coverage across the whole map that automatically keeps clear of roads, buildings, and water. All primitive placeholders — no final art yet — but re-running the command gives an entirely different, fully-connected zone layout each time.
- **Textured ground:** the terrain is no longer flat gray — it's painted with grass, dirt, and rock, generated procedurally (no imported textures) and blended automatically based on the same layout data as everything else: worn dirt naturally shows up around roads, buildings, and water, and bare rock shows up on steep slopes, all keyed off the actual generated layout rather than being hand-painted.
- **The zone is now built at real-world scale — ~28-30km edge to edge**, matching the actual Chernobyl Exclusion Zone (and the reference map's own scale bar), not an arbitrary test size. This forced a real technical change: individual trees/bushes/grass as separate objects (last entry's approach) is physically impossible at this scale — sparse coverage alone would need millions of objects, which would hang the Editor and blow up the save file. Foliage now uses Unity's built-in GPU-instanced Terrain systems instead (still fully procedural, still no imported art) — same clearance logic keeps it off roads/buildings/water either way. Road/river winding, the boundary fence, settlement sizes, and ruin counts were all retuned so object counts stay reasonable at the new scale.
- **A "hero" landmark apartment building, modeled on real Pripyat photos the team shared** — a 9-floor block with actual windowed facades (repeating window rows, not one plain wall), balconies from the 2nd floor up, a stairwell entrance bay with a canopy, panel seams and a base plinth, a rooftop water tank and a stylized rooftop-signage silhouette, and a genuinely furnished, walkable ground floor: a dormitory room with bed frames, a room with a tilted cabinet/mirror/torn curtains and scattered debris (peeling-wallpaper vibe), a bathroom with a real red/yellow checkerboard tile pattern and a tub, and a lobby with a stairwell railing and support pillar. Everything is still primitives — no imported meshes or textures. This is deliberately a one-off showcase, not the template for every building in a settlement grid (see below).
- **Fixed: the building read as one flat gray slab and the furniture was nearly invisible.** The first pass technically had all this detail, but every surface — walls, floor, and furniture alike — used nearly the same neutral gray with no interior lighting, so it all blended together into a flat mass regardless of how much geometry was actually there. Now: two alternating facade panel tones, dark contrasting balcony railings (with more posts so they read as a real barrier), a stairwell accent stripe and entrance canopy to break up the flat frontage, each interior room has its own distinctly colored back wall and floor (matching the reference photos' yellow dormitory / green peeling-wallpaper look) instead of sharing the exterior's gray, furniture recolored for contrast against the room instead of blending into it, and a point light in every room so the furnished interior is actually visible rather than rendering near-black.
- **A real procedural-modeling step up, matching much more detailed reference art the team shared.** Two new capabilities, both still 100% code-generated (no imported art): (1) actual custom mesh generation — torn/curled paper debris and peeling wallpaper strips are genuine jagged, faceted geometry now, not flat tinted cubes, and pipes/wire clutter are continuously curved tube meshes following a path rather than chains of straight cylinders; (2) procedurally-generated wall/floor *textures* (stains, cracks, wood plank seams, peeling-patch patterns baked into small generated images) applied via real materials, not flat color tints. Also added a proper multi-step staircase and tangled wire clutter in the lobby, and fixed a real bug found while building this: several props (bed frames, cabinet, mirror, tub, debris) were positioned right at the room's y=0 baseline without accounting for the floor slab + floor panel's combined ~0.12m thickness, so they were sinking into the floor — thin debris chips especially would have rendered fully submerged and invisible.

### Not started yet (rest of Phase 1)

- Footstep sounds, animations, and any visible player body model (currently invisible to itself, as expected for first-person)
- Actual hiding spot dressing (bushes, lockers, dark corners) — right now a hiding volume is just a plain dark green placeholder box

### Known issues / things to watch

- **Distance judgment is a simplification.** The "sees with both eyes vs. one eye" system captures *whether* the AI has a confident fix on the player, but it doesn't yet calculate true depth/distance the way real stereo vision does. Good enough for the prototype; may need revisiting for Phase 2 AI behavior tuning.
- **Only tested against one placeholder wall.** The vision/occlusion system hasn't been stress-tested against real level geometry yet — that happens once environment art starts going in.
- **The noise level the player generates isn't heard by anything yet.** It's calculated and exposed, but no AI system reads it until hearing/Looming behavior is built in Phase 2.
- **No jump**, even though the input binding exists — not part of this pass, wasn't asked for, skipped to avoid scope creep.
- **Confirmed working:** hiding volumes now correctly drop the player off the AI's radar the instant they step in, verified in-editor with the Predator watching. (They didn't at first — the original version relied on Unity's trigger-enter/exit events, which silently require a Rigidbody that our CharacterController-based player doesn't have. Rebuilt on the same "poll for who's nearby" approach the AI vision system already uses.)
- **If a rig ever stops responding to a new feature** (e.g. hiding suddenly "does nothing" again), the likely cause is that the Player/AI object in your scene predates the editor tool being updated to add that feature's component. Re-running the relevant `Preyed Upon` menu command now patches missing components onto an existing rig instead of silently doing nothing, so try that first.
- Debug logging is on by default for the AI vision component during this prototyping stage — expect some console chatter when testing; that's intentional for now and will be dialed back later.
- **The cave is an entrance marker, not a walkable interior.** It's a rock mound with a dark opening to mark where a cave *will* go — modeling an actual explorable cave interior is bigger scope than this blockout pass.
- **Tree/bush prototypes have no distance LOD or billboarding yet.** They're simple primitive composites rendered at full detail regardless of distance — fine functionally, but tens/hundreds of thousands of instances without LOD is a real performance concern for a shipped build, not just a cosmetic one. Adding LOD groups to the prefabs is the natural next step once real art replaces the primitives anyway.
- **A single Terrain object at 30km is a simplification, not a scalable long-term architecture.** Real large open-world games tile/stream terrain in chunks; we're using one big Terrain, which works (Unity's single-precision floats are fine at this range) but means coarse texture/detail resolution everywhere (dirt paths and grass patches are chunkier up close than they'd be with a smaller, higher-resolution terrain) and no ability to unload distant parts of the map from memory.
- **Generation now takes noticeably longer** (several seconds instead of near-instant) — expected at 30km instead of 300m, not a hang. Give it a moment after clicking the menu command.
- **Settlements are still a token representation, not a to-scale city.** The grids got bigger to better fill out the real zone size, but a real Pripyat has dozens of large apartment blocks across several km — ours is a much smaller stand-in, not a 1:1 reconstruction.
- **Trees/bushes don't block AI sightlines yet** (only walls, fences, and the cave rocks are on the AI's Occlusion layer). Foliage-based partial occlusion is a reasonable Phase 2 follow-up, not attempted here.
- **Buildings all face the same direction** (their doorway always opens toward one fixed side) since placement doesn't randomize rotation yet — a cosmetic gap, not a functional one.
- **POI positions are "rough spot + jitter," not fully free placement.** The reactor and each settlement have an intended rough location relative to the zone center (matching the reference map's general arrangement), randomized within a radius of that spot — not placed completely freely, so every seed produces a layout with the same overall shape (reactor west-ish, big settlement northeast-ish, etc.) with the details varied.
- **The reference map's glowing "anomalies" weren't built.** That's a STALKER-style mechanic in the reference image that isn't part of this game's design docs — flag it if you actually want that as a gameplay feature (radiation-style hazard zones are already on the Phase 3 roadmap separately) or if it was just meant as map flavor to ignore.
- **Roads are a minimum spanning tree, not a road hierarchy.** Every POI is guaranteed reachable with no redundant loops, but real road networks usually have a "main road" and branching side roads — this pass treats every connection the same width/style.
- **Ground textures are speckled solid colors, not photographic ground textures.** Three layers (grass, dirt, rock), each a small procedurally-generated tinted texture — reads fine at blockout distance, but is a placeholder for when real ground textures come in. Water edges (river/pond) get the same "dirt" layer as roads/buildings rather than a distinct muddy bank texture, to keep this pass to three layers instead of four. At real zone scale the texture resolution is also fairly coarse (~29m per texel) — dirt paths will look a bit chunky/blocky up close rather than following the road edge smoothly, an inherent limit of one terrain covering 30km (see below).
- **The landmark apartment isn't wired into the settlement generator yet.** It's a standalone building you place with its own menu command right now, not automatically dropped into the Pripyat-style grid — swapping 1-2 of the grid's plain apartment blocks for this one is a natural next step if wanted.
- **Fixed: the landmark apartment's upper 8 floors were hollow behind the windowed facade.** Now furnished the same way generic zone buildings are: each floor is split into one unit per window bay (skipping the stairwell bay), and each unit picks its own random RoomType via RoomFurnisher and gets furnished accordingly, alternating between two real wall materials per unit for variety. Only the ground floor keeps its original fixed, hand-detailed 4-room layout (dormitory/wallpaper room/bathroom/lobby) with the genuine custom-mesh debris -- that's a deliberate showcase, not something the random system replaces.
  - Room type currently has no bearing on whether a unit's size can plausibly fit what got placed in it (same caveat as the generic-building room system).
- **A real staircase now connects all 9 floors, and doorways actually let you walk between rooms.** Previously the upper floors had no way to be reached at all (the lobby's staircase was a short decorative flight going nowhere), and every partition wall between rooms/units was completely solid with no opening. Now:
  - Every partition wall -- the ground floor's 4-room split and every upper-floor unit boundary, including the one next to the stairwell bay -- has a doorway gap instead of sealing the room off. 60% of doorways get a real hinged door (an `InteractableDoor` component tracking open/closed state and exposing `Open()`/`Close()`/`Toggle()`, physically swinging around its hinge edge); the rest stay open gaps, which fits the abandoned setting as well as a broken-open door would.
  - **Known caveat:** nothing calls `Open()`/`Close()`/`Toggle()` yet -- there's no player-facing interaction system wired up, so doors are physically real and toggleable in code but not yet something the player can actually open by walking up to one.
- **Fixed: the first staircase pass was one straight 14-step flight rising the entire floor height (~35 degrees) and didn't reliably work for an actual player.** Individual 0.2m-rise steps sat right at the edge of a `CharacterController`'s default step-offset, with no continuous surface underneath to fall back on. Rebuilt as a proper switchback (dog-leg) staircase matching a real stairwell -- two parallel half-flights around a central void, each rising half the floor height, joined by a landing at the turn, with simple railings along the open edge and a matching opening cut into the floor above so each flight actually passes through. Each half-flight also gets an invisible sloped ramp collider spanning its full rise/run underneath the visual tread steps -- the incline is well under the default 45 degree slope limit, so this guarantees a human can actually climb it regardless of how the discrete tread colliders behave on their own. Both half-flights enter/exit at the same Z position every floor, so where you arrive on one floor is exactly where the next flight up begins.
- **Fixed: partition walls (the ground floor's 4-room split and every upper-floor unit boundary) never had a real material at all -- just a flat tinted color, the one wall type in the whole building that got missed when everything else moved to real textures.** Now textured the same way as every other real wall (per-piece tiling included): the ground floor's partitions use the neutral exterior-facade material (each of those 4 rooms already has its own distinct wall material, so the partition between them stays neutral rather than picking a side), and upper-floor unit partitions alternate between the same two materials the units themselves use.
- **Fixed: the stairs weren't actually reachable through the doorway meant to connect to them.** Two compounding bugs: (1) the per-floor slab excludes the entire stairwell bay column so both flights have room to pass through, but the only solid floor inside that column was the two flight footprints and the landing -- everywhere else (including right where you'd want to step off the stairs to reach a doorway) was open air over nothing. Added solid floor strips filling the rest of the bay column around the flight openings, so the arrival point at the top of each flight now has real floor connecting it outward. (2) The doorway into the unit next to the stairwell was still using the generic room-center Z position every other partition's doorway defaults to, which has nothing to do with where the stairs actually enter or exit -- it could land you facing a random point mid-flight instead of the landing. That doorway now aligns specifically with the stairs' actual entry/exit Z (shared between the two builder methods via one `StairEntryZ` helper, so they can't drift out of sync).
  - **Fixed: that doorway was placed exactly at the flight's own edge, which is also where the railing starts, so it was still partially blocked.** Moved the doorway from `StairEntryZ` to a new `StairDoorwayZ`, further out in the same solid margin strip (see above) but clear of both the flight and its railing.
- **Fixed: the real "stretched concrete wall" bug, project-wide.** `FacadeBuilder.ComputeTileScale` (the shared per-piece tiling fix everything real-textured routes through) picks the panel's two largest dimensions as its tiling basis, but the (u,v) *order* it returned didn't actually match Unity's own default cube UV mapping per face -- top/bottom and front/back panels happened to get it right, but any wall thin in **X** (the building's own left/right exterior side walls, and the partition walls added a couple of passes ago) had its U and V repeat counts swapped. A wall that's short in Y and long in Z -- exactly what an exterior side wall or a long partition segment is -- would show only the *short* dimension's repeat count stretched across the *long* dimension, reading as a badly stretched texture. Since this lives in the one shared helper, the fix applies everywhere real materials are used: the landmark apartment's exterior/partition walls **and** every generic zone building's walls (settlements, checkpoints, ruins, reactor), all of which route through the same code.
- **The rooftop "signage" is a stylized row of blocks, not literal readable text.** The real Pripyat rooftop slogan is Cyrillic lettering; spelling actual letters out of primitives was out of scope here, so it's a suggestive silhouette instead.
- **Fixed: torn paper and wallpaper strips were rendering solid black.** The double-sided-mesh safety net (rendering both triangle windings so nothing risked being invisible from a backwards normal) had a real bug: both windings shared the same vertices, and `RecalculateNormals()` averages the normals at each shared vertex — two exactly-opposite face normals average out to a near-zero vector, which renders as pure black under lighting no matter what color the material actually is. Fixed by giving each winding its own separate vertex copies instead of sharing indices, so the normals no longer cancel out. Still double-sided (each shape has 2x the triangles it strictly needs), just correctly so now.
- **The landmark apartment's debris/wallpaper placement isn't seeded** — it uses its own unseeded random generator, so re-running the placement command gives a different debris arrangement each time. Fine for a standalone showcase; would need a seed parameter if this ever gets wired into the main zone generator for reproducible layouts.
- **Fixed: wall textures had an obvious tile-grid look.** One shared Material was being reused across every wall piece (piers, sills, lintels, end walls — all different sizes), but Unity's default cube UVs are always 0-1 per face regardless of world size, so a 0.3m pier and a 12m wall both showed exactly the same number of texture repeats — visibly inconsistent seams between adjacent pieces. Fixed by giving each piece its own small material instance with tiling scaled to that piece's actual size, so texel density now stays consistent across the whole facade instead of look like a patchwork of differently-stretched tiles.
- **The new wall/floor textures are small (128px) and generic** — one damaged-concrete pattern, one wood floor pattern, three wallpaper tints, each generated once and reused everywhere. Real production texturing would want more variation and higher resolution; this is a procedural placeholder step up from flat color, not a final texture pass.
- **First deliberate exception to "no imported art": a real PBR texture set on the wallpaper walls.** The team supplied an actual Albedo/Normal/Smoothness texture set for wall surfaces, specifically to fix the tile-seam issue with a real texture instead of a bigger procedural one. It's imported with correct settings (the normal map is flagged as a Normal Map on import, not treated as plain color data), and since URP's Lit shader has no separate smoothness slot — it expects smoothness baked into an alpha channel — the smoothness image gets baked into the albedo texture's own alpha channel once, and the resulting material reads it from there. This material replaces the procedural wallpaper on the two rooms that originally showed the tiling problem (the dormitory and the peeling-wallpaper room); the bathroom tile and lobby concrete stay procedural for now. Falls back to the old procedural wallpaper automatically if the texture files aren't present, so the building tool still works standalone.
  - **Fixed: two of those three source files weren't actually valid PNGs.** `normal.png` and `smoothness.png` turned out to be WebP-encoded data saved with a `.png` extension, which Unity's PNG importer can't read (threw "File could not be read" and then crashed downstream). Re-encoded both as real PNGs; the importer now also checks for a failed/missing texture and logs a clear error instead of crashing if this happens again.
- **The team added a much larger real PBR texture library — seven full 4K sets (diffuse/normal/roughness, some with height/spec maps too)** — and they're now wired in everywhere they have an obvious thematic match to an existing procedural surface, each with automatic fallback to the procedural version if its files are ever missing:
  - **Terrain ground layers:** `forest_ground_06` → grass, `rocky_trail` → the dirt/worn-path layer (same use as before: shows up near roads/buildings/water), `rocky_terrain_02` → the rock layer on steep slopes. Diffuse and normal are used directly; since a full per-pixel roughness mask isn't worth the cost for ground viewed mostly from a distance, smoothness is instead a single value sampled (a coarse grid, not every pixel) from each set's roughness map.
  - **Building materials:** `damaged_plaster` → the dormitory room wall, `worn_mossy_plasterwall` → the peeling-wallpaper room wall, `weathered_brown_planks` → the interior wood floor, `gravel_concrete` → the exterior facade and lobby. These get the same real per-pixel treatment as the original wallpaper PBR set (roughness baked into the albedo texture's alpha channel, since they're seen up close).
  - **Not used yet:** the height/displacement maps (no parallax/tessellation setup exists to consume them) and one spare specular map on the rock set (this project uses the metallic/roughness workflow, not specular) — left in the folder unused rather than deleted, in case a later pass wants them.
  - **Repo size note:** this is ~578MB of new binary texture data, on top of what these bake into (similarly large combined textures/materials get generated alongside them). Committed as-is for now rather than setting up Git LFS or excluding them — flagging here since it's a big, one-way jump in repo size that's worth knowing about if clone times start to matter.
- **The team added a real 3D prop library — 22 modeled furniture/tool/weapon/loot/environment pieces with full diffuse/normal/roughness/metal texture sets, plus a couple of alpha-cutout masks for straps and device screens.** This is the project's first real *modeled* (non-primitive) content — a bigger exception to "everything starts as primitives" than the wall/ground textures were, since these are actual sculpted meshes (an old bed frame, a painted wooden cabinet, a dining table and stool, an electric stove and microwave, a coffee cart, a bolt-action rifle, an ammo box, a treasure chest, vintage suitcases, a chemistry set, a laptop, a cassette player, a radio transceiver, tools, a dead tree trunk, and more). Each one gets imported with correct texture settings and a real material built from its texture set (normal map flagged correctly, roughness inverted into smoothness, metal maps wired in properly this time via a `_MetallicGlossMap` — separate from the base texture's own alpha channel — specifically so props with an actual alpha-cutout mask, like the rifle's strap or the microwave's door window, still get real transparency instead of that channel being eaten by smoothness data). Two of the 24 texture sets couldn't be used: `industrial_storage_cart` has textures but no mesh file, and `grass_medium_01` has no mesh at all (see below).
  - **Where they ended up:** loot containers now show a real mesh where one fits thematically — a Safe is a treasure chest, a WeaponsCrate is a metal toolbox, a Bag is a vintage suitcase (randomly one of two skins) — with loose tool/weapon/electronics clutter scattered near about half of all containers for extra found-item detail. Ruins now sometimes contain a real piece of leftover furniture instead of standing empty. A sparse, hand-placed scatter of dead tree trunks was added across the zone (separate from the GPU-instanced bulk forest) as a rare, thematic detail. The landmark apartment's dormitory beds and wallpaper-room cabinet now use the real meshes too, falling back to the old primitive versions if the real prefabs aren't available for any reason.
  - **grass_medium_01** (a real grass card diffuse+alpha image) replaced the small procedural grass-blade silhouette used for Terrain's billboard grass detail layer — same fallback pattern as everything else here.
  - **Repo size note:** this adds ~1.4GB of prop meshes/textures on top of the ~578MB above — committed as-is again, same reasoning as before, but the jump is bigger this time and worth knowing about if repo size or clone time becomes a real problem later.
- **Fixed: the forest was completely invisible.** `Tree_Prototype`/`Bush_Prototype` built their trunk/foliage as child primitives with no renderer on the root object itself — Unity's Terrain tree-instancing system silently rejects a prototype prefab like that ("no valid mesh renderer") and places nothing, with only a console warning to notice by. Fixed by combining the child primitives into one mesh directly on the root (still no imported geometry, just merged submeshes so each part keeps its own tinted material) so the root itself has a real renderer. This means every previous run of the wilderness generator had an empty forest regardless of tree/bush density settings — re-running the generator now should show it.
- **Fixed: `weathered_brown_planks` silently fell back to the procedural wood floor.** Every other 4K PBR set uses `.exr` for its roughness map except this one, which exported as `.jpg` — the hardcoded `.exr` path meant the file was never found, and it fell back to the procedural material without erroring loudly (just a console warning). Fixed by making the roughness extension configurable per texture set.
- **Improved: multi-material props (coffee cart, bolt-action rifle, radio transceiver) now match texture sets to mesh parts by keyword instead of raw slot order.** These three props have two separate texture sets for two different parts of the same mesh; slot-order assignment was putting the wrong texture set on most of the mesh. Now each submesh's original FBX-assigned material name is matched against a keyword (e.g. "access" for the accessories set) and falls back to the main-body texture set for anything that doesn't match. Still not perfect — flagging if a texture set still looks wrong on one of these three, since the keyword guess isn't verified against the actual FBX data.
- **Fixed: the real bed mesh was lying on its back instead of standing upright.** `old_bed_frame_4k` was authored lying flat, not matching the primitive bed's upright convention — added the missing 270° X rotation so it stands correctly.
- **Fixed: real loot-container meshes (reported: the weapons crate/metal toolbox) could sink into the terrain.** Every real FBX prop comes with whatever pivot its source file happened to be authored with — some sit on their base, some are centered on their bounding box — so placing everything at exactly ground-level Y assumed a base pivot that isn't true for all of them. Real loot container visuals, ruin furniture, and dead trees now measure their actual rendered bounds after rotation/scale and shift so the bottom sits exactly at ground level, regardless of the source mesh's own pivot convention.
- **Fixed: the loose clutter items scattered next to loot containers (reported: something sunk into a wall corner) never got the same ground-snap treatment as the containers themselves** — a straight miss, now fixed the same way.
- **Fixed: loot containers could spawn clipping into a nearby building wall.** They were jittered within only 6m of a settlement/reactor/checkpoint's center, which is tight enough to reliably land inside a building's own footprint. Widened to 18m — this isn't a real per-building collision check (no footprint data exists to check against), just enough room that landing inside a wall becomes the exception rather than the norm.
- **`Locker` loot containers now use the real worn metal rack mesh.** `Dumpster` still has no real-prop equivalent and stays a primitive.
- **The bathroom's back wall now uses the real gravel-concrete facade material** instead of the procedural `NeutralWallpaper` — no texture set thematically fits a plain bathroom wall specifically, so this reuses the same material the lobby already does, leaving no procedural wall material left in the landmark building except the checkered tile pattern (a deliberate stylized pattern, not a placeholder) and the primitive props with no real-mesh equivalent (bathtub, leaning mirror, curtain remnant).
- **Every generic building in the zone now uses the real building materials too, not just the landmark apartment.** Checkpoint booths, the reactor complex, every settlement apartment block/village shack, and every ruin were still flat-tinted primitive walls (the original per-building "everything starts as primitives" approach) even after the real materials existed elsewhere. `BuildingKit` (and `SettlementGenerator`, which calls into it) now take real Material references and apply the same per-piece tiling fix the landmark building's facade uses, falling back to the old flat color if no material is given. Since there are potentially hundreds of these buildings across a 30km zone -- far more repetition than the one landmark building -- each building type alternates between two of the four real materials (picked per-building, not fixed) so a big settlement grid or 50 scattered ruins don't all show the exact same texture:
  - Reactor: gravel-concrete (walls, floor, stack).
  - Dense settlement (apartment blocks): gravel-concrete / worn-mossy-plasterwall alternating, gravel-concrete floor.
  - Village settlements + checkpoint booths: damaged-plaster / worn-mossy-plasterwall alternating, wood-plank floor.
  - Ruins: damaged-plaster / worn-mossy-plasterwall alternating, wood-plank floor.
  - **Performance note:** each wall/floor/roof piece now gets its own small Material instance (needed for correct per-piece tiling, same technique as the landmark building), so a full zone generation now creates on the order of thousands of individual Material objects rather than reusing a handful of cached flat-color materials. Editor generation time and runtime memory footprint both go up as a result -- not measured precisely, but worth knowing about if generation starts feeling noticeably slower or heavier than before.
- **Random room-type generation: every generic building interior now picks what kind of room it is and furnishes itself accordingly, instead of standing as an empty shell or getting one random piece of furniture.** New `RoomType` enum (Bedroom, Bathroom, Office, Kitchen, Storage, EmptyRuin) and a `RoomFurnisher` that places a type-appropriate mix of the real props for whichever type gets picked:
  - Bedroom: bed frame, sometimes a cabinet.
  - Bathroom: the primitive bathtub (no real bathroom prop exists yet), sometimes a lighter as clutter.
  - Office: a table/stool as a desk setup, plus 1-2 pieces from a pool of laptop/radio/chemistry-set/spacecraft-instrument/cassette-player.
  - Kitchen: a dining table, 1-3 stools, and a chance each of a stove, microwave, and coffee cart.
  - Storage: 2-4 pieces from a pool of toolbox/rack/ammo box/tools/suitcases.
  - EmptyRuin: deliberately close to bare -- only a 30% chance of exactly one leftover piece, meant to read as "picked clean" rather than a recognizable furnished room.
  - Every settlement shack and every floor of every apartment block now picks its own type independently (an intact building is always furnished as *something* -- EmptyRuin is reserved for actual ruins, which additionally get a 35% chance of rolling it instead of a normal type). A `RoomInfo` marker component tags each room with the type it got, for whatever gameplay system wants to query it later (nothing does yet). This is a first pass -- room type currently has no bearing on wall/floor material (that's still picked independently per building), and there's no guarantee a room's size actually fits what got placed in it.
  - **Fixed: props in the same room could land right on top of or immediately next to each other** (reported: a coffee cart's espresso machine practically touching the dining table). Every prop placement in a room now retries a handful of random candidates and keeps whichever ends up furthest from everything already placed in that room -- not a real bounding-box collision check (props still have no actual size awareness), just enough of a spacing heuristic to stop the obvious overlap.
  - **Fixed: appliances with a visible cord (stove, microwave, coffee cart) were placed freestanding anywhere in the room instead of against a wall like something plugged into an outlet would be.** These three (plus the bed frame and cabinet, which are just as unrealistic floating in the middle of a room) now use a wall-placement path: pick one of the room's four walls, sit near it with a small inset margin, and face outward from it, instead of a fully free/random point and rotation.
- **Settlement apartment blocks now have a real staircase connecting every floor, same as the landmark apartment.** Room generation was already wired into settlements, but each floor of a generic `BuildApartmentBlock` was still a fully disconnected box stacked on the one below it -- no way to actually climb between floors at all. Extracted the landmark apartment's switchback-staircase code (two half-flights around a void, a landing, railings, an invisible sloped ramp collider per half-flight for guaranteed `CharacterController` climbability, solid floor strips around the flight openings) into a shared `StaircaseBuilder`, so both builders use the exact same tested implementation instead of two copies that could drift apart. Each apartment block's stairwell sits in one corner, away from the ground-floor entrance centered on the front wall.
  - **Performance note:** a dense settlement is up to 48 apartment blocks, most of them 4 floors (3 staircases each) -- this adds a real amount of extra geometry (treads, ramp colliders, railings, floor strips) across a whole settlement, on top of the per-piece Material instancing noted above. Not measured precisely, but worth watching if a dense settlement's generation time or object count becomes a real problem.
- **The dense settlement now uses the actual full landmark apartment for every building, not the generic apartment block.** This is a real step up from "same kind of building, more detail" -- every position in the dense settlement grid is now a genuine 9-floor building with real furniture, doors, and a connecting staircase per unit, not a simplified stand-in. Given how much heavier that is than a plain box, the grid shrunk from 8x6 (48 buildings) down to 4x3 (12) -- confirmed with the team first, since generating 48 full landmark buildings risked minutes-long generation or an Editor hang. Spacing between buildings went from 20m to 50m to fit the landmark's actual 36m width (the old spacing was sized for the generic apartment block's 10m default), and the settlement's clearance/flatten-pad radius grew to match the bigger real footprint. `BuildingKit.BuildApartmentBlock` (with its own staircase, from the previous pass) is no longer called by anything, but kept as-is rather than deleted in case a simpler mid-density building type is wanted later. Village settlements are unaffected -- still small single-story shacks.
- **Removed the zone boundary fence and both water features (river, pond).** The team asked for these gone outright, not replaced -- `BuildBoundaryFence`, `BuildRiver`, and `BuildPond` (plus the now-unused `RiverGenerator.cs` and `PropFactory.CreateWaterPlane`) are deleted entirely rather than just skipped, so there's no dead code left calling into them. The reactor's own perimeter fence is untouched (a different, thematically-important feature -- it still rings the reactor complex via the same `FenceBuilder` the boundary fence used, which is why `FenceBuilder.cs` itself is still around). The zone boundary polygon (`Result.BoundaryPolygon`) is also still computed and still used to place the two checkpoint gates -- only the fence *along* it is gone.
- **Fixed: every street light in the zone pointed the same fixed world direction regardless of the road next to it.** They were offset from their road waypoint by a constant `(3.5, 0, 0)` world-space vector and built with no rotation at all, so on a road running north-south the lamp arm would stick out sideways into or away from the road at random depending on which way the road happened to run. `BuildRoadNetwork` now computes each waypoint's road tangent from its neighboring waypoints, offsets the light along the *perpendicular* of that tangent (so it sits roadside relative to the road's actual local direction, not a fixed compass direction), and rotates it to face back toward the road. `PropFactory.CreateStreetLight` takes an optional rotation now and its arm/lamp-head were rebuilt along local +Z instead of +X specifically so a plain `Quaternion.LookRotation(towardRoad, Vector3.up)` works directly at the call site with no extra correction angle needed. Wrecked cars got the same tangent-relative treatment (previously a fixed `(-4, 0, 2)` world offset), placed on the opposite roadside from the lights instead of at a fixed compass bearing.
- **Real asphalt roads and a two-tier road hierarchy, closer to the reference map's look.** The team added a `worn_asphalt` 4K PBR set (`Assets/_Project/Props`); `RoadGenerator` now builds a real per-piece-tiled material from it (same tiling-by-piece-size technique as walls, since road segments vary hugely in length) instead of the old flat brown color, falling back to that color if the texture files are ever missing. Roads are also no longer all the same style: any road touching one of the two major hubs (the reactor or the large dense settlement) is now a wider (6m), straighter "primary" road; everything else (checkpoint/village-only connections) stays a narrower (3m), wavier "secondary" track -- a rough approximation of the reference map's thick through-roads vs. thin side tracks, not a full road-class graph.
- **A real, much denser mixed-species forest, replacing the single procedural cone-and-sphere tree.** The team added three real tree species (`pine_tree_01`, `fir_tree_01`, `pine_sapling_medium`), each with a real bark (trunk) material and a separate alpha-cutout twig (needle-cluster) material, wired in through the same FBX+PBR pipeline as every other real prop. `VegetationInstancer` now takes an array of tree species instead of one and picks a random species per placed instance, so the forest doesn't read as one tree repeated everywhere; falls back to the old single procedural tree if none of the three are available. Density was also pushed way up (tree cell size 40m→18m, chance 10%→55%; bush cell size 30m→15m, chance 12%→30%) to read as actual forest coverage instead of a scattered blockout placeholder. Each species' extra `trunk_a`/`trunk_b`/`trunk_c` texture variants aren't used yet -- wiring per-instance trunk variation would need a bigger change than another material slot, so they're left in the folder unused like the height/displacement maps elsewhere.
  - **Known risk, unverified: the three tree FBX files are enormous** (256MB/307MB/957MB on disk, almost certainly from embedded 4K textures across bark+twig+the unused trunk variants, not necessarily mesh complexity alone) -- this couldn't be checked in-Editor from here. At 55% tree chance across a 30km zone this could mean a very long generation time, a bloated scene file, or a real framerate/memory problem once GPU-instanced at density. Please run the generator and watch for a hang or a bad framerate; if it's a problem, the likely fixes are dropping tree chance back down, leaning on `pine_sapling_medium` for bulk coverage and reserving the two full trees for sparser accents, or asking for lower-poly/smaller versions of these three files.
- **Fixed (reported: the dense forest made everything very laggy) without reducing tree count or density at all:** two real, independent causes, both rendering-side rather than placement-side. (1) Every real Material this project builds (walls, props, terrain-adjacent, and now tree bark/twig) was leaving `enableInstancing` at its default `false` -- without it, Unity can't batch identical trees into GPU-instanced draw calls and instead submits one draw call per tree, which is catastrophic once placement counts reach the hundreds of thousands. Now set to `true` everywhere a material gets built (`RealPBRTextureUtil.BuildMaterial`/`BuildFullMaterial`, `PropFactory.GetMaterial`, `WallTextureSetup`); per-piece material copies (walls, road segments) inherit it automatically since they're built by copying an already-fixed base material. (2) None of the three real tree species ship a billboard/LOD asset, so every placed tree -- no matter how far away -- was being rendered at full mesh detail with no cheaper fallback. `VegetationInstancer` now sets the Terrain's tree rendering distance to a capped 300m (with a 450-tree max-full-detail count) right after placing instances -- trees beyond that just aren't drawn, they're still fully present in the data and will reappear as you approach. Both fixes are purely about *how* the already-placed trees get rendered, not how many exist.
  - **Since these two fixes wrote the corrected `enableInstancing`/tree-distance settings into newly-generated Materials and Terrain data, but every real Material/prop prefab already on disk had been baked by the old code and every `Ensure*` helper just reuses a file if it already exists, the fixes alone changed nothing for content already generated.** Deleted every previously-generated real material/prefab (confirmed with the team first, since it's a real bulk deletion of ~90 tracked files across `Prefabs/Props`, `Prefabs/PropTextures`, `WallTextures`, `RoadTextures`, and the tree/bush fallback in `Vegetation`) so they rebuild correctly the next time the generator runs -- all of it is 100% procedurally regenerated, nothing hand-authored was touched.
- **Fixed (reported: still laggy, this time in the Editor Scene view itself rather than Play mode) -- the real remaining cause was sheer GameObject count, not rendering.** Every wall panel, floor slab, roof cap, road segment, and bathroom tile is (and stays) its own `GameObject.CreatePrimitive` cube -- none of that is GPU-instanced the way the forest is, so 12 full 9-floor landmark buildings, 50 furnished ruins, every settlement/checkpoint, and the whole road network were landing in the tens of thousands of individual GameObjects, each with its own Editor overhead (bounds, gizmos, hierarchy/undo tracking) independent of GPU rendering cost. New `GeometryCombiner` is a purely additive post-pass (no changes to FacadeBuilder/BuildingKit/LandmarkApartmentBuilder/RoadGenerator themselves) that walks an already-built hierarchy and merges wall/floor/roof/road-segment/tile pieces sharing the same parent (each floor/room/road already parents its own pieces, so this lands at exactly the right granularity) and the same underlying texture-or-color into one combined mesh + one MeshCollider, cutting GameObject count by roughly the piece count per floor/room/road. The tricky part: every piece currently carries its own tiling scale on a per-piece Material copy (`mainTextureScale`), not in the mesh itself, so combining naively would collapse many different tile scales down to one -- instead each piece's tiling gets baked directly into a copy of its own UVs before combining, so the merged result can share one plain material and still look pixel-identical to before. Props, doors, stair treads/ramp colliders, and every other named/decorative piece are left completely untouched (the combiner only touches GameObjects named exactly `WallSegment`/`WallPanel`/`Wall`/`Floor`/`Roof`/`RoadSegment`/`Tile`), and layer is part of the grouping key alongside parent+texture specifically so a floor slab and its walls -- which can share the exact same facade material -- never get merged into one object and silently lose the wall's Occlusion layer.
- **A held, functioning rifle: idle sway, movement bob, bolt-action firing with recoil and a muzzle flash, and a reload.** The team asked for "hyper realistic" weapon animation, which isn't achievable here -- there's no first-person arms rig or animation clips in the project, only the existing `bolt_action_rifle_7_62` prop mesh (previously just world-placed loot/clutter), and building genuinely realistic animation needs an actual animated asset the team would have to source, not something proceduralizable from code. Went with fully code-driven weapon handling instead, confirmed with the team as the right tradeoff for now:
  - New `Assets/_Project/Scripts/Weapons/` (`WeaponSway`, `MuzzleFlash`, `RifleWeapon`) plus an Editor `WeaponSetup` tool (`Preyed Upon/Phase 1/Attach Rifle To Player`) that parents the rifle prefab under the player's camera as a viewmodel.
  - `WeaponSway` owns the weapon's transform every frame -- idle breathing sway, a footstep-timed bob scaled by the player's actual speed/sprint state, and a reload "dip" pose -- all blended toward smoothly so stance changes and sprint start/stop don't snap. Recoil is a `Kick()` impulse that self-recovers on top of whatever the current idle/bob pose is, rather than a separate animation fighting for the same transform.
  - `RifleWeapon` is the input/state side: a 5-round bolt-action magazine (one shot per trigger pull, not automatic), a cooldown matching bolt-action cadence, and a timed reload. Firing also kicks the camera itself upward via a new `PlayerLook.AddRecoilKick` (a temporary, self-recovering offset layered on top of the player's actual aim, not a permanent change to it, so the view settles back to wherever you were actually aiming) and triggers `MuzzleFlash` (a Light spike plus a small unlit emissive quad -- no imported flash texture, consistent with this project's procedural-first approach for anything that isn't an explicitly supplied asset).
  - **Real bug caught before it shipped:** `PlayerInput`'s `SendMessages` notification mode only calls handler methods on the exact GameObject the `PlayerInput` component sits on (`Player`), not on descendants -- and the rifle viewmodel lives several levels down the camera hierarchy. `PlayerRigBuilder` now uses `BroadcastMessages` instead (reaches `Player` and every descendant), with the existing "patch an already-built rig" path in the same tool updated to fix this on a Player built by an older version too, not just fresh ones.
  - Added a `Reload` input action (`R`, plus Gamepad d-pad up) -- `Attack` already existed in the input actions asset (bound to left mouse) but nothing used it until now.
  - **Two real, unverified guesses that will very likely need a visual check in the Editor:** the viewmodel's screen position/rotation (a generic FPS offset guess, not measured against the actual mesh) and the muzzle point (derived from the mesh's own bounding box, assuming the model's local +Z is the barrel direction -- true for most character-facing assets, but unconfirmed for this specific one). If the rifle sits wrong on screen or the flash doesn't line up with the barrel, both are just plain child transforms to nudge by hand, not something to regenerate.
  - No damage/health system exists yet (Phase 2 AI behavior hasn't started) -- a shot that hits something just logs what it hit, as a placeholder for whenever real combat consequences get built.
  - **Fixed (reported: fire/reload still did nothing after attaching the rifle): the `BroadcastMessages` fix above only takes effect on a Player rig that gets rebuilt through `PlayerRigBuilder` after pulling it -- an already-built Player left on the old `SendMessages` setting would silently never call the rifle's input handlers.** `RifleWeapon` now forces `PlayerInput.notificationBehavior` to `BroadcastMessages` itself in `Awake`, so this works regardless of whether the rig tool gets re-run.
- **A highly-irradiated-zone atmosphere: dense fog, dim overcast lighting, and a sickly color-graded look.** New Editor tool (`Preyed Upon/Environment/Setup Irradiated Atmosphere`) -- deliberately separate from Build Wilderness Blockout, since lighting/fog/post-processing are scene-wide settings, not part of any specific zone layout, so regenerating the zone doesn't wipe this out. Sets up: linear fog (clear up to 15m, fully fogged out by 140m -- easier to tune at real zone scale than an exponential density constant) in a hazy olive-gray; a dim, cool-toned directional sun at a steep overcast angle rather than a warm low-angle one, paired with a dark flat ambient so shadowed areas don't go pure black; and a global URP Volume with a new `IrradiatedZoneVolumeProfile` (desaturated ~35%, a faint sickly green-yellow color filter, a dark vignette, and subtle bloom so lights/the muzzle flash read a little hazy rather than perfectly crisp). Also flips on the player camera's Post Processing flag, which URP leaves off by default and which the Volume does nothing without.
  - All of it is plain tunable values in `AtmosphereSetup.cs` -- re-running the command reapplies the fog/light/camera settings immediately, but the Volume Profile asset itself is only built once (checked via `AssetDatabase.LoadAssetAtPath`), so delete `Assets/_Project/Data/IrradiatedZoneVolumeProfile.asset` first if you change the color-grading numbers and want a re-run to pick them up.
- **A main menu and in-game pause menu, for the first playtest build.** Two new Editor tools: `Preyed Upon/UI/Build Main Menu Scene` creates a standalone `MainMenu` scene (Start Game / Exit Game) and registers both it and the gameplay scene in Build Settings (menu first, so a build actually boots into it); `Preyed Upon/UI/Add Escape Menu To Scene` adds a Resume / End Game / Exit Game pause menu (toggled with Escape) to whichever scene is currently open -- run that one in the gameplay scene. Both use plain legacy uGUI (`Text`/`Image`/`Button`, not TextMeshPro) built entirely from code, specifically so there's no font-asset-import step needed for a first playtest.
  - `MainMenuController`/`EscapeMenuController` wire their own button clicks in `Start()` rather than relying on serialized UnityEvent listeners, so the Editor tools only have to assign which `Button` goes to which field.
  - The escape menu freezes the player by disabling its movement/look/weapon components directly, not just via `Time.timeScale = 0` -- `PlayerLook`'s mouse-look isn't frame-rate scaled, so a nonzero look delta at the exact moment of pausing would otherwise keep spinning the camera every `Update()` even at timescale 0 (`Update` still runs; only `Time.deltaTime` becomes 0). `RifleWeapon`'s fire/reload handlers also got an explicit `enabled` check for the same reason from the other direction -- a message-dispatched handler like `OnAttack` runs regardless of the component's `enabled` state (unlike `Update`, which Unity gates automatically), so disabling the component alone wouldn't actually have blocked a shot from firing while paused without that check.
  - **Real bug caught before it shipped:** building the main menu scene additively (so it doesn't discard whatever scene you already had open, including unsaved zone-generation work) means it briefly coexists with your current scene in the Editor. A naive "does an EventSystem already exist" check searches *all* loaded scenes, not just the new one -- it would have found your gameplay scene's EventSystem and skipped creating one for MainMenu, leaving that scene without its own EventSystem (and therefore non-functional buttons) once loaded standalone in a real build. `MenuUIBuilder`'s EventSystem check is now scene-scoped instead of project-wide.
  - Build Settings scene registration only knows the gameplay scene by name (`SampleScene`, both tools' `gameplaySceneName` field) -- if it's ever renamed, re-run `Build Main Menu Scene` after updating the constant, or fix it directly in Build Settings.
  - **Fixed: the team renamed the gameplay scene from `SampleScene` to `GamePlayScene`, so Start Game had nowhere valid to load.** Updated the hardcoded name in both `MainMenuSceneBuilder` and `MainMenuController`'s default -- re-running `Build Main Menu Scene` picks up the fix and (re-)registers the correctly-named scene in Build Settings.
  - **Fixed: `GamePlayScene.unity` is 119.88MB, over GitHub's 100MB push limit** -- the fully-generated 30km zone serializes into a lot of scene data. Added `*.unity` to Git LFS tracking alongside the existing `*.fbx` rule, same fix as the earlier oversized tree assets.
- **Fixed (reported: still laggy, asked to only render what's in view of the player) -- the camera's far clip plane was still Unity's 1000m default.** With fog fully hiding everything past 140m, Unity was still frustum-culling, sorting, and rendering geometry (buildings, the dense forest, whole settlements) many times further out than anything actually visible -- real GPU/CPU cost for zero visual benefit. `AtmosphereSetup` now caps the camera's far clip plane to just past the fog line (160m), and `VegetationInstancer`'s tree render distance was tightened to match (160m, down from 300m) so Terrain's own CPU-side tree culling doesn't keep considering trees the camera can't render anyway. Also switched the camera's background from Skybox to a solid color matching the fog -- without that, anything beyond the new clip plane would show sky instead of more fog, a visible seam right where the ground recedes into the distance.
- **Fixed: a real Player build failed outright with "Tree prefab at index 3 is missing" (2 errors), even though everything looked and ran fine in the Editor.** Index 3 is the bush prototype (indices 0-2 are the three real tree species) -- its prefab reference had gone null, almost certainly a stale reference in the saved scene left over from an earlier deleted-and-regenerated `Bush_Prototype` that the scene wasn't re-saved against at the time. The Editor tolerates a null `TreePrototype.prefab` silently during Play; a Player build validates every one and fails the whole build if any is null. `VegetationInstancer.PlaceTrees` now filters out null tree/bush prefab references before building the prototype list (degrading to "one less species" instead of blocking every build) and no longer assumes species indices line up 1:1 with the original prefab array once nulls are filtered.
  - **Practical fix if this recurs:** re-run Build Wilderness Blockout, then explicitly save the scene (Ctrl+S) before building a Player -- a build always packages the last-*saved* scene state, not whatever's live in the current Editor session.
  - Also fixed two `FindFirstObjectByType` obsolescence warnings (`EscapeMenuController`, `MenuUIBuilder`) flagged by this Unity version in favor of `FindAnyObjectByType` -- functionally identical here (exactly one Player/EventSystem per scene), just without the ordering guarantee `FindFirstObjectByType` provides and pays for.
- **Fixed (reported: a real build no longer errored, but crashed outright the instant Start Game loaded the gameplay scene).** `Player.log` showed Unity's own engine reporting the level file "corrupted" followed by `[Position out of bounds!]` and a hard native crash -- happened identically on every attempt, so it was deterministic bad data in the saved scene, not a random fluke. That specific message is Unity's engine-level guard against a NaN/Infinity (or absurdly large) value passing through as a Transform or mesh bounds, which the Editor tolerates silently but a Player build's stricter validation hard-crashes on with no indication of which object caused it.
  - New `Preyed Upon/Debug/Find Invalid Transforms` walks every loaded scene and reports the full hierarchy path of anything with an invalid position/rotation/scale (or, since a corrupted Transform isn't the only way to get here, an invalid `Renderer`/`MeshCollider` bounds) -- run it against the *current* scene before regenerating anything, or the evidence gets overwritten.
  - Also hardened `GeometryCombiner` defensively while investigating: it computes `parent.worldToLocalMatrix * piece.localToWorldMatrix` per combined piece, and inverting a transform with a zero (or near-zero) world scale on any axis produces exactly this kind of NaN/Infinity, baked straight into a combined mesh's vertices with no visible symptom on the GameObject's own (perfectly normal-looking) Transform. Both the shared parent's and each individual piece's world scale are now checked before combining; a degenerate one is excluded (with a warning) instead of ever producing invalid geometry, whether or not this turns out to be the actual cause here.
  - **Still crashed identically after a fresh regeneration + rebuild, even though the diagnostic tool above found nothing wrong.** The gap: `TreeInstance` data (a tree's placement on a Terrain) lives inside `TerrainData` as a plain array, completely separate from the GameObject/Transform hierarchy the tool was walking -- a NaN `TreeInstance.position` (itself normalized 0-1 within the terrain) wouldn't show up as a broken Transform anywhere, but is exactly what the engine's "position out of bounds" wording describes literally. `Find Invalid Transforms` now also scans every `Terrain`'s tree instances for an invalid position/scale/rotation/prototype index.
  - **Real bug found in the same pass:** `VegetationInstancer.CollectInstances`'s own range check (`if (normX < 0f || normX > 1f || ...) continue;`) silently fails to catch a NaN `normX`/`normZ` -- every comparison against NaN is `false` in IEEE 754, so a NaN value passes this guard as if it were perfectly in-range instead of being rejected. Added an explicit `float.IsNaN` check ahead of the range comparison.

## 2026-07-12 — The gameplay-scene crash is finally fixed, plus a real inventory system

### Where we're at

The deterministic crash on Start Game (reported at the end of the last entry) is resolved. Also new: a working inventory system with a real item catalog, displayed diegetically on an in-world laptop prop instead of a flat menu overlay.

### Done

- **The root cause of the "[Position out of bounds!]" crash, finally found: a script component defined in the wrong file.** Every previous fix (removing an oversized combined road mesh, cutting tree density, dropping an unusably high-poly real tree asset) turned out to be treating symptoms, not the cause -- each one changed *when* the crash happened, not whether it happened. The actual break: Play Mode reproduced the exact same scene with zero issues every time, which meant the problem was specific to how a **Player build** serializes the scene, not the scene's actual content. `RoomInfo` (a small marker component tagging every furnished room with its type) lived inside `RoomFurnisher.cs`, a file whose main type is a differently-named static class -- violating Unity's one-file-per-`MonoBehaviour` convention, which apparently doesn't matter for Play Mode's live in-memory type resolution, but corrupts something in build-time script-to-GUID serialization when hundreds of instances of it accumulate across the zone (one per furnished room/ruin/shack). Moving it into its own file fixed it outright. Also fixed along the way: `terrain.detailObjectDistance` was never explicitly set, so grass was popping out of existence unfogged at its default ~80m cutoff, well short of where the fog actually goes fully opaque (140m) -- now matched to the fog distance so grass fades into it instead. Tree/bush density was also trimmed as a first pass at reported lag, and bushes were dropped entirely (low-poly enough to add little visual value at this density while still costing a full extra instance-placement/culling pass).
- **A one-click `Preyed Upon/Build/Regenerate Zone, Save, and Build And Run` tool.** Nearly every round of this crash investigation traced back to the same mistake: regenerating the zone or editing a generator script without also re-saving the scene before building -- a Player build always packages the last-*saved* scene, not whatever's live in the Editor. This does regenerate + save + build as one atomic action so that particular mistake isn't possible anymore.
- **First-person arms, attempted -- currently on hold.** The team sourced a real arms model (wriks' "WRAD ARMS," CC0) and got as far as attaching it as a rigid viewmodel (parented under the rifle so it inherits all of `WeaponSway`'s existing motion -- idle sway, bob, recoil, reload dip -- for free, the same "no dedicated animation clips" tradeoff the rifle itself already made). The model shipped with no bones initially, then was re-exported with a full hand/finger rig and a hand-authored holding pose -- but rebuilding the cached prefab from the new export blew away the pose (a prefab rebuild always re-instantiates fresh from the FBX's current rest pose; there's no mechanism to preserve a pose that only exists in the Editor scene, not the source asset, across a re-export). Shelved for now; the `Attach Arms To Player` tool still works whenever a properly re-posed-and-exported version is ready.
- **A real inventory system, styled as an old CRT terminal readout and displayed on an in-world laptop prop.** Built from a reference image the team shared (a green monochrome "ZONE 7 HARVEST LOG" terminal table). Two real data classes -- `ItemDefinition` (an item's fixed identity: name, category, flavor text, and which real prop backs it, if any) and `PlayerInventory` (the mutable held stack: quantity, condition, location, notes) -- plus a `ItemDatabase` cataloging all 150 items from the team's full design list across five categories (Tools & Navigation, Weapons & Defenses, Consumables, Crafting & Loot, Folklore & Anomalous Artifacts), each with its real flavor text. Only six items currently have a real prop matched to them (Prybar/`Crowbar`, Screwdriver, Multitool/`Pliers`, Walkie-Talkie/`VintageRadioTransceiver`, Hunting Rifle/`BoltActionRifle`, Lighter/`VintageLighter`) -- everything else is real, structured data sitting ready for whenever a prop gets gathered for it, rather than a rewrite.
  - Pressing Tab brings a `ClassicLaptop` prop into view in front of the camera (same viewmodel trick as the rifle/arms), with the inventory table rendered as a World Space canvas parented directly to the laptop's screen -- a real 3D object in the world showing the UI, not a flat overlay. Escape and Tab are mutually aware so pausing while the inventory is open closes the inventory first rather than the two fighting over whether the player's movement/look/weapon should be frozen.
  - **Real bug caught before it shipped:** the controller polling for the Tab key was originally going to live on the laptop object itself -- but that object starts inactive (hidden until opened), and Unity never runs `Update()` on components under an inactive GameObject, so Tab would have silently done nothing. It lives on the always-active Player instead, just holding a reference to the laptop it toggles.
  - **Real bug caught before it shipped:** the same escape-menu freeze logic (and the equivalent inventory one) only disabled the rifle's *script component*, not its GameObject -- which stopped firing/reloading but left the rifle mesh visibly rendering (and clipping through the pause menu / inventory laptop) the whole time either was open. Both now disable the whole rifle viewmodel GameObject.
  - The laptop's held position and its screen's exact placement/rotation/scale on the mesh were pure guesses at first (no source dimensions for the prop) and needed several rounds of hands-on tuning in the Editor, including one full 180-degree flip on both the laptop and the screen canvas (a World Space canvas's readable side faces its own local -Z, so nesting it under a 180-flipped parent needs a second 180 to cancel back out) -- now baked into the setup tool as the working defaults.

## Roadmap (high level)

**Phase 1 — Prototype** *(in progress)*
Core AI vision, basic player movement/hiding, and a rough blockout environment to test in.

**Phase 2 — AI Behavior & Phenotypes** *(not started)*
Predators methodically search areas in sweeping rows; prey wander randomly and skittishly react to noise. AI gets "looming" and "peeking" behaviors around corners. Environment gets dressed with debris and abandoned building interiors.
